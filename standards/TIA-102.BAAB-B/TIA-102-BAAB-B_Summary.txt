TIA-102.BAAB-B: Conformance Tests for the Project 25 Common Air Interface
Summary

DOCUMENT OVERVIEW

TIA-102.BAAB-B is the conformance test specification for the Project 25 (P25) Phase 1 FDMA
Common Air Interface (CAI). Published March 4, 2013 (revision B-B), it provides the
normative test vectors and informative reference C source code required to verify that a
P25 implementation correctly encodes every data unit type defined in TIA-102.BAAA. This
document is targeted at equipment manufacturers and test labs seeking TIA P25
Compliance Assessment Program (CAP) certification.

PURPOSE AND SCOPE

The P25 CAI defines the over-the-air format for digital land mobile radio interoperability.
Because P25 uses a chain of forward error correction (FEC) codes, the final bit pattern
output for any given set of voice or data parameters is fully deterministic. TIA-102.BAAB-B
captures this deterministic output in exhaustive test vector tables, enabling implementors
to validate each encoder stage in isolation or as an integrated whole.

The test coverage spans every normative data unit type in P25 Phase 1 FDMA:
- Header Data Unit (HDU, DUID=0x0)
- Simple Terminator (DUID=0x3)
- Link Data Unit 1 (LDU1, DUID=0x5) — carries voice and Link Control
- Link Data Unit 2 (LDU2, DUID=0xa) — carries voice and Encryption Sync
- Confirmed and Unconfirmed Data Packets (DUID=0xc)

FRAME STRUCTURE

Each P25 data unit begins with a 24-bit Frame Sync (FS) constant:
  {0x5575, 0xf5ff, 0x77ff, 0x0000}
followed by a 64-bit Network ID (NID) containing the 12-bit Network Access Code (NAC),
4-bit DUID, and 48 BCH(64,16,23) parity bits. The remainder of each data unit carries
its specific payload (voice channel, link control, encryption sync, or data blocks)
interleaved with 2-bit Status Symbols (SS) that indicate radio idle state.

FEC CODING CHAIN

The P25 CAI applies a layered FEC strategy:

  NID:              BCH(64,16,23) — corrects up to 11 bit errors in the 64-bit NID
  Header:           RS(24,12,13) over GF(2^6) — 12 six-bit symbols in, 24 out
  Link Control:     RS(24,16,9) over GF(2^6) — 16 symbols in, 24 out
  Encryption Sync:  RS(36,20,17) over GF(2^6) — 20 symbols in, 36 out
  IMBE u0–u3:       Golay(23,12) — 12 info bits to 23-bit code words
  IMBE u4–u6:       Hamming(15,11,3) — 11 info bits to 15-bit code words
  Post-RS output:   Hamming(10,6,3) — 6 info bits to 10-bit code words
  Low Speed Data:   Cyclic(16,7) — 16-bit CRC-style protection
  Data packets:     Trellis 3/4 and 1/2 rate convolutional codes + CRC-CCITT/CRC-32

IMBE VOICE ENCODING DETAILS

LDU1 and LDU2 each carry 9 IMBE voice frames. Each IMBE frame provides 8 u-vectors
(u0 through u7), 12 bits each, for a total of 72 u-bits per frame. The coset code for
u1–u3 is generated by a linear feedback shift register:
  p = (173 * p + 13849) & 0xffff, seeded from 16 * u[0]
This LFSR deterministically covers u1–u3 to improve error performance.
The voice channel bits are interleaved across the LDU frame using a 72×4 schedule.

TEST VECTOR STRUCTURE

All test vectors are expressed as microslot output. A microslot contains 6 twelve-bit
words (3 hex digits each), displayed in column pairs (two microslots per line). LDU frames
produce 24 microslots (microslots 0, 2, 4, ..., 22). Data packets produce 4 to 18
microslots depending on block count.

Test vectors are organized by:
- Data unit type (B.1 HDU through B.7 Data Packets)
- Radio configuration: Case 1 = Subscriber Radio (SSym=all zeros),
                       Case 2 = Repeater Radio (SSym=all ones)
- NAC value: 12 values covering diverse 12-bit patterns
  (0x023, 0x031, 0x032, 0x043, 0x114, 0x205, 0x40b, 0x411, 0x464, 0x516, 0x585, 0x811)
- Link control parameters: LCF, MFID, TGID, Sourc, Dest, Emerg across wide value ranges

The LDU1/LDU2 parameter sweeps in sections B.3–B.5 are exhaustive: source addresses
step through all 24-bit powers of two (0x000001, 0x000002, 0x000004, ..., 0x800000),
destination addresses similarly, and LCF takes values 0x00 and 0x03. This ensures every
bit of the Link Control field is exercised through the RS(24,16,9) encoder.

REFERENCE IMPLEMENTATION (ANNEX A)

Annex A (informative) contains the original C source code written circa 1993 by Al Wilson,
Jim Holthaus, LeCocq, and ALW that generated all test vectors. Five programs are provided:
hdr.c (Header DU), term.c (Terminator), ldu1.c (LDU1), ldu2.c (LDU2), and data.c (Data
Packets). These programs accept parameters on stdin and write microslot hex to stdout,
enabling reproducible generation of additional test cases beyond those listed in Annex B.

CONFORMANCE REQUIREMENT

Equipment under test shall produce microslot output bit-exactly matching the Annex B
vectors for the given input parameters. Any deviation in FEC encoding, field ordering,
interleaving, or status symbol placement indicates a non-conformant implementation.

RELATIONSHIP TO OTHER TIA-102 DOCUMENTS

TIA-102.BAAB-B is a test companion to TIA-102.BAAA (the normative CAI definition).
The IMBE vocoder input u-vectors are defined in TIA-102.BABA. Security fields (MI,
ALGID, KID) are defined in TIA-102.AABF. Together these form the Phase 1 FDMA air
interface stack: vocoder → CAI framing and FEC → RF symbol mapping.

IMPLEMENTATION NOTES

The P25 CAI FEC chain is entirely over GF(2^6) for the Reed-Solomon codes and GF(2)
for the binary block and convolutional codes. The BCH(64,16,23) NID code is the outermost
protection layer; an SDR receiver that fails BCH decoding cannot identify the NAC or DUID
and must discard the frame. The trellis codes for data packets use a 16-state convolutional
structure and interleave output dibits using a fixed 98-entry rule before CQPSK mapping.

CLASSIFICATION NOTE

Per the TIA-P25 processing pipeline, this document is classified CONFORMANCE. It does not
define new algorithms; it validates implementations of algorithms defined elsewhere.
No Phase 3 implementation specification is produced for conformance documents.
