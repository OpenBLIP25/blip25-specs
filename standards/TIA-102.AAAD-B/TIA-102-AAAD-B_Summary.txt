TIA-102.AAAD-B: Project 25 Digital Land Mobile Radio Block Encryption Protocol
Summary — approximately 1,000 words

WHAT THIS DOCUMENT SPECIFIES

This document defines the Block Encryption Protocol (BEP) for Project 25 (P25) digital public safety Land Mobile Radio (LMR) systems. It specifies precisely how block cipher algorithms must be applied to protect both voice and data communications transmitted through the P25 Common Air Interface (CAI). The protocol operates above the Layer 2 functions defined in the FDMA CAI (BAAA) and the Two-Slot TDMA MAC (BBAC), functioning as an upper sublayer of the data link layer or as Layer 3/4 processing.

The document was first published as an Interim Standard in March 2001, incorporating AES-256 and Triple-DES alongside the original DES. It was formally published as a full standard in July 2002, updated with a Phase 2 Two-Slot TDMA addendum, revised as TIA-102.AAAD-A in August 2009, and published as TIA-102.AAAD-B in December 2015 following a five-year ANSI renewal. No patents are identified as applying to this standard.

WHY IT EXISTS

P25 public safety systems serve police, fire, EMS, and other agencies that frequently transmit sensitive information over shared radio spectrum. Encryption is mandated as a capability requirement in the TIA-102 suite. Without a precisely defined protocol governing how encryption algorithms interact with the CAI superframe structure, interoperability between different manufacturers' equipment would be impossible — an encrypted transmission from a Motorola radio must be decryptable on a Harris console. This document provides that interoperability glue, specifying not just which algorithms to use, but the exact bit-level details of how keystream is generated and applied to every octet position in every voice frame and data packet.

WHAT IT DEFINES

The protocol has four key components: (1) a block cipher algorithm with associated key variable, (2) an operating mode specifying how the algorithm generates keystream, (3) an initialization vector that uniquely seeds each transmission, and (4) a Message Indicator (MI) mechanism that enables mid-stream synchronization recovery.

SYNCHRONIZATION (MI/LFSR): The MI is a 64-bit value transmitted in the CAI header and periodically throughout each voice message. It seeds a 64-bit Linear Feedback Shift Register (LFSR) with characteristic polynomial C(x) = 1 + x^15 + x^27 + x^38 + x^46 + x^62 + x^64. Each superframe advances the LFSR by 64 clocks to produce the next MI, creating a maximal-length sequence of period 2^64 - 1. The periodically embedded MI allows a receiver to resynchronize mid-message — critical for public safety dispatch where call beginnings are frequently missed. The MI uses 64 of the 72 CAI-allocated bits (bits 8-71); bits 0-7 are reserved and set to zero.

KEYSTREAM GENERATION (OFB MODE): All supported algorithms are operated in Output Feedback (OFB) mode. The LFSR state is loaded into the block cipher input register (with expansion for algorithms where n > 64 bits), then the cipher is clocked once to produce an encrypted MI — this first output is discarded and fed back as the new input. Subsequent iterations produce keystream blocks of m = floor(n/8) octets each (any rightmost r = n mod 8 bits are discarded from each output but still fed back). This OFB chaining provides the property that a single channel bit error causes only one plaintext bit error.

VOICE OPERATION — FDMA: Each FDMA voice superframe contains 213 encryptable octets: 3 reserved + 8 LC + 18 IMBE frames (11 octets each) + 4 Low Speed Data octets. For DES/TDEA (n=64, m=8), 28 OFB iterations are needed per superframe (B = 1 + ceil(213/8) = 28). For AES-256 (n=128, m=16), 15 iterations are needed. The LC field carries the rightmost 64 bits of the 80-bit Link Control word. IMBE voice words (u0-u7) are packed into 11 octets (w0-w10) per frame. Tables 5-5 and 5-6 give the complete encryption schedule mapping each plain text octet to its OFB block and intra-block position for both LDU1 and LDU2.

VOICE OPERATION — TWO-SLOT TDMA: For Phase 2 P25 (TDMA), each 360 ms interval contains 18 half-rate vocoder voice frames. Each frame yields 7 octets (49 bits rounded up), giving 126 encryptable octets per interval. For DES (n=64), 17 OFB iterations are needed; for AES-256 (n=128), 9 iterations. The half-rate vocoder packs words u0-u3 into 7 octets (W0-W6); the least significant 7 bits of W6 are not transmitted. The MI is carried in the ESS-A/ESS-B distributed encryption sync fields defined in BBAC.

DATA OPERATION: Data messages are fragmented and packaged as packets per BAAA. An Encryption Synchronization (ES) auxiliary header is prepended to each fragment, carrying 9 MI octets (octets 0-7 carry MI bits 71-8 respectively; octet 8 is reserved = $00), 1 ALGID octet, 2 KID octets, and 1 secondary SAP octet. Everything after the ES header is encrypted except the CRC and any headers preceding the ES. OFB operation follows the same procedure as for voice: MI is encrypted first (result discarded but fed back), then data octets are XORed with successive keystream blocks.

KEY PARAMETERS AND RESERVED VALUES: The Key ID (KID, 16 bits) identifies which key to use; KID $0000 is the reserved default for single-key systems. The Algorithm ID (ALGID, 8 bits) identifies the cipher; ALGID $80 is reserved for unencrypted, $81 for DES, $83 for TDEA, and $84 for AES-256 (all per BAAC). An all-zero MI is reserved as invalid and indicates unencrypted operation — any IV generation procedure producing all-zero output must retry.

SUPPORTED ALGORITHMS: Three normative annexes specify algorithm-specific details. Annex A covers DES: n=64, k=64 (every 8th bit is parity), ALGID=$81, using 64-bit OFB. Annex B covers Triple-DES (TDEA): n=64, k=192 (every 8th bit parity), ALGID=$83, operation E_K3(D_K2(E_K1(input))), using TOFB64 mode. Three-key TDEA with K1=K2=K3 is backwards compatible with DES. Annex C covers AES-256: n=128, k=256, ALGID=$84, 14 rounds, using 128-bit OFB. For AES, since n=128 > 64, the MI must be expanded using the LFSR shift register technique, producing a 128-bit input register value from successive LFSR states.

TEST VECTORS: The document provides extensive worked test vectors for all three algorithms in both FDMA and TDMA configurations. The canonical test MI is 1234 5678 90AB CDEF 00 (from Table 3-1). DES uses key 0123 4567 89AB CDEF. TDEA uses key bundle K1=0123 4567 89AB CDEF, K2=2345 6789 ABCD EF01, K3=4567 89AB CDEF 0123. AES-256 uses a 256-bit key split across two 128-bit halves. Full plaintext/ciphertext pairs are given for complete voice superframes (FDMA) and 360ms TDMA intervals.

RELATIONSHIPS TO OTHER DOCUMENTS: This document is tightly coupled to BAAA (FDMA CAI, which defines the superframe structure and MI field placements), BBAC (Two-Slot TDMA MAC, which defines ESS placement), BAAC (reserved values for ALGID and KID), and BABA (vocoder description, which defines the IMBE word structure packed into w0-w10). It is compatible with NIST FIPS 197 (AES), FIPS 46-3 (DES), and ANSI X9.52 (TDEA). The encryption protocol is explicitly NOT part of the CAI; it operates above Layer 2 and may be implemented in devices not directly connected to the CAI.

LIMITATIONS: The document does not specify how the Initialization Vector is generated — this is left to manufacturers. It does not address key management, key distribution, or over-the-air rekeying. It does not specify authentication or message integrity. AES-128 and AES-192 are not addressed, only AES-256. DES is included for legacy compatibility but offers inadequate security by modern standards.
