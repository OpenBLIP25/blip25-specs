# P25 Link Layer Authentication -- Implementation Specification

**Source:** TIA-102.AACE-A (April 2011)
**Classification:** ALGORITHM + PROTOCOL
**Phase:** 3 -- Implementation-ready
**Extracted:** 2026-04-12
**Purpose:** Defines the challenge-response authentication protocol for P25 trunked radio
systems. Covers how subscriber units (SUs) authenticate to infrastructure (FNE) and
optionally how infrastructure authenticates back to subscribers (mutual authentication).
This spec provides everything a decoder/parser needs to identify, extract, and interpret
authentication messages on the air interface.

**Cross-references:**
- TIA-102.AABC-E -- Trunking Control Channel Messages (TSBK/MBT wire formats for auth messages)
- TIA-102.AABF-D -- Link Control Word formats (LCW opcodes; auth uses control channel only, not LCW)
- TIA-102.BBAD-A -- TDMA MAC Messages (auth messages mapped into TDMA partition 0b01 MAC PDUs)
- TIA-102.AAAD-B -- Block Encryption Protocol (encryption complements auth: auth protects access, encryption protects content)
- SDRTrunk / OP25 -- Open-source decoder implementations

## 1. Authentication Architecture -- What Gets Authenticated

### 1.1 Entities and Directions

P25 link-layer authentication involves three logical entities:

```
+--------------------+          +--------+          +---------+
| Authentication     |  RS,KS   |  RFSS  |  RAND1  |   SU    |
| Facility (AF)      |--------->| (site) |-------->|  (radio)|
| (holds master K)   |  KS'     |        |<--------|         |
+--------------------+          +--------+  RES1    +---------+
```

- **SU (Subscriber Unit):** The radio. Holds authentication key K (128-bit) and its
  SUID (56-bit = WACN[20] + System ID[12] + Subscriber ID[24]).
- **RFSS (RF Sub-System):** The site/base station. Does NOT hold K directly. Receives
  session keys KS and KS' from the AF.
- **AF (Authentication Facility):** Backend infrastructure. Holds K per SUID. Derives
  session keys KS and KS' using a random seed RS and distributes them to the RFSS.
  K never transits the air interface.

### 1.2 Three Authentication Modes

| Mode | Direction | Messages | When Used |
|------|-----------|----------|-----------|
| Unit Authentication | FNE challenges SU | AUTH_DMD + AUTH_RESP | FNE-initiated, non-mutual |
| Mutual Authentication | Both challenge | AUTH_DMD + AUTH_RESP_M + AUTH_FNE_RESP + AUTH_FNE_RST | SU configured for mutual auth |
| SU-Initiated Authentication | SU demands FNE start | AUTH_SU_DMD then normal flow | FNE declined to challenge during registration |

### 1.3 Relationship to Encryption

Authentication and encryption are complementary but independent:
- **Authentication (AACE):** Protects network access -- proves the SU/FNE is who it claims to be.
- **Encryption (AAAD):** Protects content -- encrypts voice/data payloads.

Authentication uses its own key hierarchy (K -> KS/KS') completely separate from
traffic encryption keys (KEK, TEK). A system can authenticate without encrypting,
encrypt without authenticating, or do both. In practice, modern P25 systems that
encrypt also authenticate.

## 2. Authentication Algorithms -- AES-Based Challenge-Response

All four authentication mechanisms use AES-128 in ECB mode (FIPS 197, NIST SP 800-38A).

### 2.1 Key Hierarchy

```
K (128-bit)                    Long-term secret, per-SUID, provisioned once
  |
  +-- AM1(K, RS) --> KS        Session authentication key (128-bit)
  |                            Used by RFSS for unit authentication (AM2)
  |
  +-- AM3(K, RS) --> KS'       Mutual session authentication key (128-bit)
                               Used by RFSS for mutual authentication (AM4)
                               KS != KS' even with same K and RS (bitwise complement)

RS (80-bit)                    Random seed, generated by AF, per session
                               Crypto-period rotation: AF changes RS periodically
                               to rotate KS/KS' without re-keying K

RAND1 (40-bit)                 Per-transaction random challenge from RFSS
RAND2 (40-bit)                 Per-transaction random challenge from SU

RES1, RES2 (32-bit each)       Truncated AES outputs -- the responses
```

### 2.2 AM1 -- Session Key Derivation

```
Input:  K (128 bits), RS (80 bits)
Output: KS (128 bits)

Step 1: Expand RS from 80 to 128 bits by zero-padding LSB side
        RS_expanded[127:48] = RS[79:0]
        RS_expanded[47:0]   = 0x000000000000

Step 2: KS = AES-128-ECB-Encrypt(key=K, plaintext=RS_expanded)
```

### 2.3 AM2 -- Unit Authentication Response

```
Input:  KS (128 bits), RAND1 (40 bits)
Output: RES1 (32 bits)

Step 1: Expand RAND1 from 40 to 128 bits by zero-padding LSB side
        RAND1_expanded[127:88] = RAND1[39:0]
        RAND1_expanded[87:0]   = 0x00000000000000000000000000 (88 zero bits)

Step 2: full_output = AES-128-ECB-Encrypt(key=KS, plaintext=RAND1_expanded)

Step 3: RES1 = full_output[127:96]  (most significant 32 bits)
```

### 2.4 AM3 -- Mutual Session Key Derivation

```
Input:  K (128 bits), RS (80 bits)
Output: KS' (128 bits)

Step 1: Expand RS from 80 to 128 bits (same as AM1)
Step 2: Bitwise complement the expanded RS
        RS_complemented = NOT(RS_expanded)
Step 3: KS' = AES-128-ECB-Encrypt(key=K, plaintext=RS_complemented)
```

The complement ensures KS' != KS even when derived from the same K and RS.

### 2.5 AM4 -- Mutual Authentication Response

```
Input:  KS' (128 bits), RAND2 (40 bits)
Output: RES2 (32 bits)

Step 1: Expand RAND2 from 40 to 128 bits (same expansion as AM2)
Step 2: full_output = AES-128-ECB-Encrypt(key=KS', plaintext=RAND2_expanded)
Step 3: RES2 = full_output[127:96]  (most significant 32 bits)
```

### 2.6 Test Vectors (from AACE-A Section 6.6)

```
AM1:
  K            = 0001 0203 0405 0607 0809 0a0b 0c0d 0e0f
  RS           = 38ae c829 33b1 7f80 249d
  Expanded RS  = 38ae c829 33b1 7f80 249d 0000 0000 0000
  KS           = 0524 30bd af39 e82f d0dd d698 c02f b036

AM2:
  KS           = 0524 30bd af39 e82f d0dd d698 c02f b036
  RAND1        = 4d 92 5a f6 08
  Expanded RAND1 = 4d925af608 00000000000000000000000000
  AES Output   = 3e00 faa8 f0c7 e864 0e92 7bb3 41b5 d44a
  RES1         = 3e00 faa8

  K            = 0001 0203 0405 0607 0809 0a0b 0c0d 0e0f
  RS           = 38ae c829 33b1 7f80 249d
  Complement   = c751 37d6 cc4e 807f db62 ffff ffff ffff
  KS'          = 69d5 dc08 023c 4652 cc71 d5cd 1e74 e104

AM4:
  KS'          = 69d5 dc08 023c 4652 cc71 d5cd 1e74 e104

  RES2         = b3ad 16e1
```

### 2.7 Security Notes

- **ECB mode is intentional:** Each authentication event is a single independent AES block.
  No IV, no chaining required.
- **32-bit RES is modest:** Only 2^32 response space. Security relies on KS secrecy
  (which requires protecting RS), not response-space exhaustion resistance.
- **RAND replay prevention:** Fresh RAND1/RAND2 per transaction plus the 30-second
  authentication timer prevent replay attacks.
- **K separation from RFSS:** K never reaches the RFSS; only KS/KS' are sent. Compromising
  an RFSS does not reveal K.

## 3. Authentication Message Formats -- Byte-Level Layouts

Authentication uses six messages: four ISP (SU -> FNE) and two OSP (FNE -> SU).
All ride on the FDMA trunking control channel as TSBKs or MBT (Multiple Block Trunking)
packets. TDMA systems with an FDMA control channel use the same formats.

### 3.1 AUTH_DMD -- Authentication Demand (OSP 0x31)

FNE challenges an SU to authenticate. This is always an MBT message (29 octets total)
because it must carry both RS (80 bits = 10 octets) and RAND1 (40 bits = 5 octets).

**Note on TSBK abbreviated form:** The AABC-E extraction shows an abbreviated TSBK format
carrying only RAND1 (32 bits in 4 octets) and a target address. This abbreviated form
does NOT carry RS. The full AUTH_DMD that initiates the AACE protocol is the MBT form
(29 octets, Header + 2 Data Blocks) which carries both RS (10 octets) and RAND1 (5 octets).
Some systems may use the abbreviated form when RS has been previously distributed; the
MBT form is the normative full authentication demand.

```
ABBREVIATED TSBK (12 octets including CRC):
  Oct 0:   LB(7) | P(6) | 0b110001 (opcode 0x31)
  Oct 1:   MFID (Manufacturer ID, 0x00 = standard)
  Oct 2:   reserved[7:1] | S[0]          S=1: standalone, S=0: within registration
  Oct 3:   RAND1[31:24]
  Oct 4:   RAND1[23:16]
  Oct 5:   RAND1[15:8]
  Oct 6:   RAND1[7:0]
  Oct 7:   reserved
  Oct 8:   Target Address[23:16]         (abbreviated, 16-bit Unit ID)
  Oct 9:   Target Address[7:0]
  Oct 10-11: TSBK CRC-16

MBT FORM (29 octets, Blocks=2):
  Header Block (12 octets):
    Oct 0:   1 | P | 0b110001           AN=1 (MBT), opcode 0x31
    Oct 1:   MFID
    Oct 2:   reserved[7:1] | S[0]
    Oct 3-7: RS[79:40]                  (first 5 octets of Random Seed)
    Oct 8-9: Target Address (abbreviated, 16-bit)
    Oct 10-11: CRC-16

  First Data Block (12 octets):
    Oct 0-4:  RS[39:0]                  (remaining 5 octets of Random Seed)
    Oct 5-9:  RAND1[39:0]              (full 40-bit Random Challenge 1)
    Oct 10-11: reserved / pad

  Last Data Block (5 octets + Packet CRC):
    Oct 0-2:  reserved
    Oct 3-4:  Packet CRC-32 (covers all blocks)
```

### 3.2 AUTH_RESP -- Authentication Response (ISP 0x38)

SU sends RES1 back to FNE (non-mutual authentication).

```
ABBREVIATED TSBK (12 octets):
  Oct 0:   LB(7) | P(6) | 0b111000 (opcode 0x38)
  Oct 1:   MFID
  Oct 2:   reserved[7:1] | S[0]        S=1: standalone
  Oct 3:   RES1[31:24]
  Oct 4:   RES1[23:16]
  Oct 5:   RES1[15:8]
  Oct 6:   RES1[7:0]
  Oct 7:   reserved
  Oct 8:   Source ID[23:16]             (abbreviated, 16-bit Unit ID)
  Oct 9:   Source ID[7:0]
  Oct 10-11: TSBK CRC-16

EXTENDED MBT (22 octets, Blocks=2):
  Header Block: WACN ID of source
  Data Block 1: WACN+System ID, Source ID, RES1 (4 bytes), S bit
  Last Data Block: reserved, Multiple Block CRC
```

### 3.3 AUTH_RESP_M -- Authentication Response Mutual (ISP 0x39)

SU sends both RES1 and RAND2 to initiate mutual authentication. This is always MBT
(27 octets) because it carries RES1 (32 bits) + RAND2 (40 bits) + S flag + full SUID.

```
MBT ONLY (27 octets, Blocks=2):
  Header Block (12 octets):
    Oct 0:   1 | P | 0b111001          AN=1 (MBT), opcode 0x39
    Oct 1:   MFID
    Oct 2-4: Source WACN ID[19:0] (packed)
    Oct 5-9: reserved / additional fields
    Oct 10-11: CRC-16

  First Data Block (12 octets):
    Oct 0-1: WACN[cont] + System ID[11:0]
    Oct 2-4: Source ID[23:0]
    Oct 5-9: RAND2[39:0]               (40-bit mutual challenge from SU)
    Oct 10-11: RES1[31:16]             (first 2 bytes of Response 1)

  Last Data Block:
    Oct 0-1: RES1[15:0]                (last 2 bytes of Response 1)
    Oct 2:   reserved[7:1] | S[0]
    Oct 3+:  reserved, Packet CRC
```

### 3.4 AUTH_FNE_RESP -- Authentication FNE Response (OSP 0x32)

FNE sends RES2 back to the SU during mutual authentication.

```
ABBREVIATED TSBK (12 octets):
  Oct 0:   LB(7) | P(6) | 0b110010 (opcode 0x32)
  Oct 1:   MFID
  Oct 2:   reserved
  Oct 3:   RES2[31:24]
  Oct 4:   RES2[23:16]
  Oct 5:   RES2[15:8]
  Oct 6:   RES2[7:0]
  Oct 7:   reserved
  Oct 8:   Target Address[23:16]
  Oct 9:   Target Address[7:0]
  Oct 10-11: TSBK CRC-16

EXTENDED MBT (16 octets, Blocks=1):
  Adds full WACN+System ID for target.
```

### 3.5 AUTH_FNE_RST -- Authentication FNE Result (ISP 0x3A)

SU tells FNE whether the FNE passed or failed mutual authentication.

```
ABBREVIATED TSBK (12 octets):
  Oct 0:   LB(7) | P(6) | 0b111010 (opcode 0x3A)
  Oct 1:   MFID
  Oct 2:   R2[7] | reserved[6:1] | S[0]
            R2=1: FNE passed authentication
            R2=0: FNE failed authentication
            S=1: standalone
  Oct 3-7: reserved
  Oct 8:   Source ID[23:16]
  Oct 9:   Source ID[7:0]
  Oct 10-11: TSBK CRC-16

EXTENDED MBT (13 octets, Blocks=1):
  Adds full WACN+System ID for source.
```

### 3.6 AUTH_SU_DMD -- Authentication SU Demand (ISP 0x3B)

SU demands that the FNE initiate an authentication challenge.

```
ABBREVIATED TSBK (12 octets):
  Oct 0:   LB(7) | P(6) | 0b111011 (opcode 0x3B)
  Oct 1:   MFID
  Oct 2-7: reserved
  Oct 8:   Source ID[23:16]
  Oct 9:   Source ID[7:0]
  Oct 10-11: TSBK CRC-16

EXTENDED MBT (12 octets, Blocks=1):
  Adds full WACN+System ID for source.
```

### 3.7 Opcode Quick Reference

| Opcode | Direction | Alias | Message | Format |
|--------|-----------|-------|---------|--------|
| 0x31 | OSP (FNE->SU) | AUTH_DMD | Authentication Demand | TSBK or MBT (29 oct) |
| 0x32 | OSP (FNE->SU) | AUTH_FNE_RESP | Authentication FNE Response | TSBK (9 oct) / MBT (16 oct) |
| 0x38 | ISP (SU->FNE) | AUTH_RESP | Authentication Response | TSBK (10 oct) / MBT (22 oct) |
| 0x39 | ISP (SU->FNE) | AUTH_RESP_M | Authentication Response Mutual | MBT only (27 oct) |
| 0x3A | ISP (SU->FNE) | AUTH_FNE_RST | Authentication FNE Result | TSBK (6 oct) / MBT (13 oct) |
| 0x3B | ISP (SU->FNE) | AUTH_SU_DMD | Authentication SU Demand | TSBK (5 oct) / MBT (12 oct) |

## 4. Non-Mutual Authentication Flow (FNE Challenges SU)

This is the most common authentication mode. The FNE challenges the SU; the SU proves
it holds the correct K-derived session key.

### 4.1 Standalone Authentication (Outside Registration)

```
    SU                                    RFSS (FNE)
     |                                      |
     |  <---- AUTH_DMD (0x31) -----------   |  FNE sends RS + RAND1
     |         [RS, RAND1, S=1, Target]     |
     |                                      |
     |  SU computes:                        |
     |    KS = AM1(K, RS)                   |
     |    RES1 = AM2(KS, RAND1)             |
     |                                      |
     |  ----- AUTH_RESP (0x38) ---------->  |  SU sends RES1
     |         [RES1, S=1, Source]           |
     |                                      |
     |  FNE compares RES1 vs XRES1          |
     |                                      |
     |  <---- ACK_RSP_FNE (0x20) --------  |  Pass: acknowledge
     |         or                            |
     |  <---- DENY_RSP (0x27) -----------  |  Fail: reason=0x22
     |         [reason="SU failed auth"]    |  "SU failed authentication"
```

### 4.2 Authentication During Registration

```
    SU                                    RFSS (FNE)
     |                                      |
     |  ----- U_REG_REQ (0x2C) ---------->  |  SU requests registration
     |                                      |
     |  <---- AUTH_DMD (0x31) -----------   |  FNE challenges before completing reg
     |         [RS, RAND1, S=0, Target]     |
     |                                      |
     |  ----- AUTH_RESP (0x38) ---------->  |  SU responds
     |         [RES1, S=0, Source]           |
     |                                      |
     |  <---- U_REG_RSP (0x2C) ----------  |  Pass: registration response
     |         or                            |
     |  <---- U_REG_RSP (extended) ------  |  Fail: REG_REFUSED
     |         [Registration=REG_REFUSED]   |
```

S=0 indicates the authentication is embedded within a registration transaction.
S=1 indicates standalone authentication (not tied to registration).

## 5. Mutual Authentication Flow (Both Sides Challenge)

Mutual authentication extends the unit authentication flow. The SU challenges the FNE
in addition to responding to the FNE's challenge.

### 5.1 Standalone Mutual Authentication -- Both Pass

```
    SU                                    RFSS (FNE)
     |                                      |
     |  <---- AUTH_DMD (0x31) -----------   |  Step 1: FNE sends RS + RAND1
     |         [RS, RAND1, S=1, Target]     |
     |                                      |
     |  SU computes:                        |
     |    KS  = AM1(K, RS)                  |
     |    KS' = AM3(K, RS)                  |
     |    RES1 = AM2(KS, RAND1)             |
     |    Generates RAND2 (40 bits)         |
     |    XRES2 = AM4(KS', RAND2)           |
     |                                      |
     |  ----- AUTH_RESP_M (0x39) -------->  |  Step 2: SU sends RES1 + RAND2
     |         [RES1, RAND2, S=1]           |           (always MBT, 27 octets)
     |                                      |
     |  FNE checks RES1 == XRES1            |
     |  If pass: RES2 = AM4(KS', RAND2)     |
     |                                      |
     |  <---- AUTH_FNE_RESP (0x32) ------   |  Step 3: FNE sends RES2
     |         [RES2, Target]                |
     |                                      |
     |  SU checks RES2 == XRES2             |
     |                                      |
     |  ----- AUTH_FNE_RST (0x3A) ------->  |  Step 4: SU reports result
     |         [R2=1 (pass), S=1]            |
     |                                      |
     |  <---- ACK_RSP_FNE (0x20) --------  |  Step 5: FNE acknowledges
```

### 5.2 Mutual Authentication -- SU Fails (RES1 Mismatch)

```
    SU                                    RFSS
     |  <---- AUTH_DMD (0x31) -----------   |
     |  ----- AUTH_RESP_M (0x39) -------->  |
     |                                      |
     |  FNE: RES1 != XRES1 --> SU FAILS    |
     |  FNE does NOT send AUTH_FNE_RESP     |
     |                                      |
     |  <---- DENY_RSP (0x27) -----------  |  reason=0x22 "SU failed auth"
```

Critical rule: when the SU fails authentication, the FNE MUST NOT send AUTH_FNE_RESP
(RES2). This prevents an adversary SU from learning RES2 and replaying it.

### 5.3 Mutual Authentication -- FNE Fails (RES2 Mismatch)

```
    SU                                    RFSS
     |  <---- AUTH_DMD (0x31) -----------   |
     |  ----- AUTH_RESP_M (0x39) -------->  |
     |  <---- AUTH_FNE_RESP (0x32) ------   |
     |                                      |
     |  SU: RES2 != XRES2 --> FNE FAILS    |
     |                                      |
     |  ----- AUTH_FNE_RST (0x3A) ------->  |  R2=0 (FNE failed)
     |  <---- DENY_RSP (0x27) -----------  |  reason=0x23 "FNE failed auth"
```

When the FNE fails mutual authentication, the SU should consider the site adversarial
and may initiate a site change to find a valid FNE.

## 6. SU-Initiated Authentication

When the FNE declines to challenge the SU during registration, a mutual-authentication-
enabled SU can demand that the FNE start authentication.

```
    SU                                    RFSS
     |  ----- U_REG_REQ (0x2C) ---------->  |
     |  <---- U_REG_RSP (0x2C) ----------  |  FNE accepts without authenticating
     |                                      |
     |  ----- AUTH_SU_DMD (0x3B) -------->  |  SU demands authentication start
     |                                      |
     |  <---- AUTH_DMD (0x31) -----------   |  FNE complies, sends challenge
     |         [RS, RAND1, S=1]             |
     |                                      |
     |  ----- AUTH_RESP_M (0x39) -------->  |  Normal mutual auth continues...
     |  <---- AUTH_FNE_RESP (0x32) ------   |
     |  ----- AUTH_FNE_RST (0x3A) ------->  |
     |  <---- ACK_RSP_FNE (0x20) --------  |
```

If the RFSS cannot authenticate the SU, it responds with DENY_RSP (0x27) with
reason code 0x24 ("SU could not be authenticated at this time").

If the FNE repeatedly declines or ignores AUTH_SU_DMD, a mutual-auth-enabled SU
transitions to the Change Site state and attempts to find a valid FNE on another site.

## 7. Key Management -- Authentication Key vs Traffic Keys

### 7.1 Key Hierarchy

```
K (128-bit)                     Authentication Key -- long-term per-SUID secret
  |                             Stored in SU non-volatile memory + AF
  |                             NEVER transmitted over the air (except via OTAR)
  |                             NEVER given to the RFSS
  |
  +-- RS (80-bit)               Random Seed -- generated by AF
  |   |                         Distributed from AF to RFSS (wireline)
  |   |                         Distributed from RFSS to SU (via AUTH_DMD over air)
  |   |
  |   +-- KS = AM1(K, RS)      Session auth key -- held by RFSS
  |   |                         Enables RFSS to compute XRES1 without knowing K
  |   |
  |   +-- KS' = AM3(K, RS)     Mutual session auth key -- held by RFSS
  |                             Enables RFSS to compute RES2 without knowing K
  |
  v
  Crypto-period rotation: AF changes RS --> new KS/KS' without re-keying K
```

### 7.2 Key Provisioning

K is provisioned into the SU and AF through one of two methods:
1. **Key Fill Device (KFD):** Per TIA-102.AACD. Physical device connects to radio.
2. **OTAR (Over-The-Air Rekeying):** Per TIA-102.AACA. K delivered over encrypted air channel.

Systems without OTAR must never transmit K over the air.

### 7.3 Separation from Traffic Encryption Keys

| Key | Purpose | Standard | Where Stored |
|-----|---------|----------|--------------|
| K (128-bit) | Authentication master key | AACE-A | AF + SU only |
| KS / KS' (128-bit) | Session auth keys | AACE-A | RFSS + SU (derived) |
| KEK (Key Encryption Key) | Protects TEK delivery | AACA (OTAR) | KMF + SU |
| TEK (Traffic Encryption Key) | Encrypts voice/data | AAAD-B | RFSS + SU |

These are completely independent hierarchies. Compromising a TEK does not reveal K.
Compromising KS does not reveal K (AES-128 ECB is not invertible without K).

## 8. Timing and Retry Behavior

### 8.1 Authentication Timer

Both the RFSS and SU start a 30-second Authentication Timer when the authentication
message sequence begins (upon sending/receiving AUTH_DMD).

```
Event                            Action
-------                          ------
AUTH_DMD sent/received           Start 30-second timer on both sides
Timer expires (no completion)    Revert to pre-authentication state
                                 RFSS: return to Idle (may retry or allow service)
                                 SU: return to Idle (may retry if SU-initiated)
```

### 8.2 Retry Behavior

**RFSS retry:** If the SU does not respond to AUTH_DMD, the RFSS may:
- Do nothing (ignore the SU), OR
- Repeat the authentication command (send AUTH_DMD again with new RAND1)

**SU retry:** If the SU does not receive R1 (pass result) from the RFSS:
- May retry the message sequence if authentication was initiated by the RFSS due to
  an SU action (like registration)

**SU site change (mutual auth):** If the FNE:
- Actively or passively declines mutual authentication
- Does not respond to RAND2 (no AUTH_FNE_RESP sent)
- Fails mutual authentication (R2 = fail)

...on one or several attempts, the SU transitions to the Change Site state and attempts
to register on a different site.

### 8.3 Service Blocking During Authentication

Rule 8 from AACE-A: an SU commanded to authenticate SHALL NOT send a separate service
request until after the authentication response has been sent. The SU must complete
(or time out of) the authentication transaction before requesting voice or data service.

On the RFSS side, if a service request (not involving authentication) is received from
a SU during Authentication Procedure in Progress, the RFSS may ignore it.

## 9. How Auth Messages Ride on Control Channel, Traffic Channel, and TDMA

### 9.1 Control Channel (FDMA TSBK/MBT)

Authentication messages are transmitted exclusively on the FDMA trunking control channel.
They use the standard TSBK and MBT framing defined in TIA-102.AABC-E.

```
TSBK frame: 12 octets total
  [LB|P|Opcode(6)] [MFID] [payload 7 octets] [CRC-16]

MBT frame: Header Block (12 oct) + N Data Blocks (12 oct each)
  Header: [1|P|Opcode(6)] [MFID] [payload] [CRC-16]
  Data:   [payload 10 oct] [CRC/pad 2 oct]
```

Message sizes (from AABC-E cross-reference):
- AUTH_DMD: 29 octets (MBT with 2 data blocks -- carries RS + RAND1)
- AUTH_RESP: 10 octets abbreviated (TSBK) / 22 octets extended (MBT)
- AUTH_RESP_M: 27 octets (MBT only -- carries RES1 + RAND2)
- AUTH_FNE_RESP: 9 octets abbreviated (TSBK) / 16 octets extended (MBT)
- AUTH_FNE_RST: 6 octets abbreviated (TSBK) / 13 octets extended (MBT)
- AUTH_SU_DMD: 5 octets abbreviated (TSBK) / 12 octets extended (MBT)

### 9.2 Traffic Channel (LCW / LDU)

Authentication does NOT use LCW (Link Control Word) messages on traffic channels.
TIA-102.AABF-D defines LCW opcodes but has only a single obsolete Unit Authentication
Command at LCO 0x12, which is reserved/obsolete. All active authentication is performed
on the control channel.

### 9.3 TDMA (Phase 2) MAC Messages

TDMA systems that use an FDMA control channel perform authentication on that FDMA
control channel using the same TSBK/MBT formats. The AACE-A specification explicitly
states: "The authentication service described in this document is applicable to FDMA
and TDMA trunking systems using an FDMA trunking control channel."

In TIA-102.BBAD-A, the authentication messages are mapped into TDMA MAC PDU partitions
for systems that may need to carry them on TDMA signaling channels:

**TDMA ISP (Inbound) -- Partition 0b01 (Abbreviated):**
| MCO | Message |
|-----|---------|
| 0x38 | Authentication Response - Abbreviated |
| 0x39 | Authentication Response Mutual |
| 0x3A | Authentication FNE Result - Abbreviated |
| 0x3B | Authentication SU Demand - Abbreviated |

**TDMA ISP -- Partition 0b11 (Extended):**
| MCO | Message |
|-----|---------|
| 0x32 | Authentication FNE Result - Extended |
| 0x38 | Authentication Response - Extended |
| 0x3B | Authentication SU Demand - Extended |

**TDMA OSP (Outbound) -- Partition 0b01 (Abbreviated):**
| MCO | Message |
|-----|---------|
| 0x31 | Authentication Demand |
| 0x32 | Authentication FNE Response - Abbreviated |

**TDMA OSP -- Partition 0b11 (Extended):**
| MCO | Message |
|-----|---------|
| 0x32 | Authentication FNE Response - Extended |

## 10. Detecting Authentication on the Air -- What a Passive Decoder Sees

### 10.1 Observable Fields

A passive decoder (no knowledge of K, KS, or KS') can observe:

```
Fully visible (cleartext):
  - AUTH_DMD:      opcode 0x31, S flag, RAND1, Target Address, RS (in MBT form)
  - AUTH_RESP:     opcode 0x38, S flag, RES1, Source ID
  - AUTH_RESP_M:   opcode 0x39, RES1, RAND2, S flag, Source SUID
  - AUTH_FNE_RESP: opcode 0x32, RES2, Target Address
  - AUTH_FNE_RST:  opcode 0x3A, R2 (pass/fail), S flag, Source ID
  - AUTH_SU_DMD:   opcode 0x3B, Source ID

Not visible (secret):
  - K, KS, KS' -- never transmitted in authentication messages
  - XRES1, XRES2 -- computed internally, never transmitted
```

### 10.2 What a Decoder Can Determine

From passive observation of authentication traffic:

1. **Which SU is being authenticated:** Target Address in AUTH_DMD, Source ID in AUTH_RESP
2. **Authentication mode:** AUTH_RESP = non-mutual; AUTH_RESP_M = mutual
3. **Standalone vs registration-embedded:** S flag (bit 0 of octet 2)
4. **Pass or fail result:** ACK_RSP_FNE after AUTH_RESP = pass;
   DENY_RSP with reason 0x22 = SU failed; R2 in AUTH_FNE_RST = FNE pass/fail
5. **Mutual auth outcome:** R2=1 in AUTH_FNE_RST means FNE passed; R2=0 means FNE failed
6. **SU-initiated auth:** AUTH_SU_DMD followed by AUTH_DMD indicates SU demanded auth start
7. **The random challenges:** RAND1 and RAND2 are visible but useless without KS/KS'
8. **The random seed RS:** Visible in AUTH_DMD MBT form; useless without K
9. **Authentication frequency:** How often the system re-authenticates radios

### 10.3 Deny Reason Codes Related to Authentication

```
0x22  "SU failed authentication"         -- RES1 mismatch, SU is likely adversary
0x23  "FNE failed authentication"        -- RES2 mismatch, FNE is likely adversary
0x24  "SU could not be authenticated"    -- RFSS lacks RS/KS, cannot perform auth
```

### 10.4 Passive Decoder Event Log Pattern

A typical authentication sequence on a busy system looks like:

```
[12:00:01.234] OSP AUTH_DMD      target=0x001234 RAND1=0xA3B7C901 S=0  (during registration)
[12:00:01.456] ISP AUTH_RESP     source=0x001234 RES1=0x7E3F0A22  S=0
[12:00:01.678] OSP U_REG_RSP    target=0x001234                        (auth passed)

[12:00:05.100] OSP AUTH_DMD      target=0x005678 RAND1=0xD1E2F3A4 S=0  (mutual auth)
[12:00:05.322] ISP AUTH_RESP_M   source=0x005678 RES1=0x11223344 RAND2=0x5566778899 S=0
[12:00:05.544] OSP AUTH_FNE_RESP target=0x005678 RES2=0xAABBCCDD
[12:00:05.766] ISP AUTH_FNE_RST  source=0x005678 R2=1(pass) S=0
[12:00:05.988] OSP U_REG_RSP    target=0x005678                        (mutual auth passed)
```

```python
def parse_tsbk_auth(opcode: int, data: bytes, direction: str) -> dict:
    """
    Parse authentication-related TSBK messages.
    data = 10-byte payload (octets 0-9 of TSBK, after stripping CRC)
    direction = "osp" or "isp"
    """
    mfid = data[1]

    if direction == "osp":
        if opcode == 0x31:  # AUTH_DMD
            s_flag = data[2] & 0x01
            rand1 = (data[3] << 24) | (data[4] << 16) | (data[5] << 8) | data[6]
            target = (data[8] << 8) | data[9]
            return {
                "type": "AUTH_DMD",
                "standalone": bool(s_flag),
                "rand1": rand1,
                "target_id": target,
                "mfid": mfid,
                "note": "abbreviated form; RS not present, check MBT for full form"
            }

        elif opcode == 0x32:  # AUTH_FNE_RESP
            res2 = (data[3] << 24) | (data[4] << 16) | (data[5] << 8) | data[6]
            target = (data[8] << 8) | data[9]
            return {
                "type": "AUTH_FNE_RESP",
                "res2": res2,
                "target_id": target,
                "mfid": mfid
            }

    elif direction == "isp":
        if opcode == 0x38:  # AUTH_RESP
            s_flag = data[2] & 0x01
            res1 = (data[3] << 24) | (data[4] << 16) | (data[5] << 8) | data[6]
            source = (data[8] << 8) | data[9]
            return {
                "type": "AUTH_RESP",
                "standalone": bool(s_flag),
                "res1": res1,
                "source_id": source,
                "mfid": mfid
            }

        elif opcode == 0x3A:  # AUTH_FNE_RST
            r2 = (data[2] >> 7) & 0x01
            s_flag = data[2] & 0x01
            source = (data[8] << 8) | data[9]
            return {
                "type": "AUTH_FNE_RST",
                "r2_pass": bool(r2),
                "standalone": bool(s_flag),
                "source_id": source,
                "mfid": mfid
            }

        elif opcode == 0x3B:  # AUTH_SU_DMD
            source = (data[8] << 8) | data[9]
            return {
                "type": "AUTH_SU_DMD",
                "source_id": source,
                "mfid": mfid
            }

    return None


def parse_mbt_auth_resp_mutual(header: bytes, data_blocks: list) -> dict:
    """
    Parse AUTH_RESP_M (0x39) -- always MBT, 2 data blocks.
    header = 12-byte header block
    data_blocks = list of 12-byte data blocks
    """
    # Extract source WACN/System/ID and RAND2 + RES1 from data blocks
    db1 = data_blocks[0]
    db2 = data_blocks[1]

    # RAND2 is 5 bytes in data block 1 (octets 5-9)
    rand2 = int.from_bytes(db1[5:10], 'big')

    # RES1 spans end of data block 1 and start of data block 2
    res1_hi = (db1[10] << 8) | db1[11]
    res1_lo = (db2[0] << 8) | db2[1]
    res1 = (res1_hi << 16) | res1_lo

    s_flag = db2[2] & 0x01

    return {
        "type": "AUTH_RESP_M",
        "rand2": rand2,
        "res1": res1,
        "standalone": bool(s_flag)
    }


def parse_mbt_auth_demand(header: bytes, data_blocks: list) -> dict:
    """
    Parse AUTH_DMD (0x31) -- MBT form with RS + RAND1.
    header = 12-byte header block (carries first 5 octets of RS)
    data_blocks = list of data blocks (remaining RS + RAND1)
    """
    s_flag = header[2] & 0x01
    target = (header[8] << 8) | header[9]

    # RS: first 5 bytes from header octets 3-7, next 5 from data block 1 octets 0-4
    rs_hi = header[3:8]      # RS[79:40]
    rs_lo = data_blocks[0][0:5]  # RS[39:0]
    rs = int.from_bytes(rs_hi + rs_lo, 'big')

    # RAND1: 5 bytes from data block 1 octets 5-9
    rand1 = int.from_bytes(data_blocks[0][5:10], 'big')

    return {
        "type": "AUTH_DMD_MBT",
        "standalone": bool(s_flag),
        "rs": rs,
        "rand1": rand1,
        "target_id": target
    }
```

### 11.2 Authentication State Tracker

```python
class AuthTracker:
    """Track authentication transactions for a passive decoder."""

    def __init__(self):
        self.pending = {}  # target_id -> {start_time, rand1, rs, ...}
        self.completed = []

    def on_auth_dmd(self, target_id, rand1, rs, standalone, timestamp):
        self.pending[target_id] = {
            "target": target_id,
            "rand1": rand1,
            "rs": rs,
            "standalone": standalone,
            "start_time": timestamp,
            "mutual": False
        }

    def on_auth_resp(self, source_id, res1, standalone, timestamp):
        if source_id in self.pending:
            self.pending[source_id]["res1"] = res1
            self.pending[source_id]["resp_time"] = timestamp

    def on_auth_resp_mutual(self, source_id, res1, rand2, standalone, timestamp):
        if source_id in self.pending:
            self.pending[source_id]["res1"] = res1
            self.pending[source_id]["rand2"] = rand2
            self.pending[source_id]["mutual"] = True
            self.pending[source_id]["resp_time"] = timestamp

    def on_auth_fne_resp(self, target_id, res2, timestamp):
        if target_id in self.pending:
            self.pending[target_id]["res2"] = res2

    def on_auth_fne_result(self, source_id, r2_pass, timestamp):
        if source_id in self.pending:
            entry = self.pending.pop(source_id)
            entry["fne_auth_pass"] = r2_pass
            entry["end_time"] = timestamp
            self.completed.append(entry)

    def on_ack_or_deny(self, target_id, passed, reason_code, timestamp):
        if target_id in self.pending:
            entry = self.pending.pop(target_id)
            entry["su_auth_pass"] = passed
            entry["reason_code"] = reason_code
            entry["end_time"] = timestamp
            self.completed.append(entry)

    def expire_stale(self, current_time, timeout=30.0):
        """Remove entries older than the 30-second auth timer."""
        expired = [k for k, v in self.pending.items()
                   if current_time - v["start_time"] > timeout]
        for k in expired:
            entry = self.pending.pop(k)
            entry["result"] = "timeout"
            self.completed.append(entry)
```

```rust
/// Authentication parameters per TIA-102.AACE-A
pub mod auth {
    /// Authentication key -- 128-bit long-term secret per SUID.
    /// NEVER transmitted over the air interface in authentication messages.
    #[derive(Clone, Zeroize)]
    pub struct AuthKey {
        pub bytes: [u8; 16],  // 128 bits
    }

    /// Random Seed -- 80-bit value generated by the Authentication Facility.
    /// Transmitted in AUTH_DMD (MBT form) from FNE to SU.
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct RandomSeed {
        pub bytes: [u8; 10],  // 80 bits
    }

    impl RandomSeed {
        /// Expand RS from 80 bits to 128 bits by zero-padding LSB side (AM1 input).
        pub fn expand(&self) -> [u8; 16] {
            let mut out = [0u8; 16];
            out[0..10].copy_from_slice(&self.bytes);
            // out[10..16] already zero
            out
        }

        /// Expand and bitwise complement (AM3 input).
        pub fn expand_complement(&self) -> [u8; 16] {
            let expanded = self.expand();
            let mut complemented = [0u8; 16];
            for i in 0..16 {
                complemented[i] = !expanded[i];
            }
            complemented
        }
    }

    /// Session Authentication Key -- 128-bit, derived via AM1(K, RS).
    /// Held by RFSS and derived by SU.
    #[derive(Clone, Zeroize)]
    pub struct SessionKey {
        pub bytes: [u8; 16],  // 128 bits
    }

    /// Mutual Session Authentication Key -- 128-bit, derived via AM3(K, RS).
    #[derive(Clone, Zeroize)]
    pub struct MutualSessionKey {
        pub bytes: [u8; 16],  // 128 bits
    }

    /// Random Challenge -- 40-bit per-transaction nonce.
    /// RAND1 from FNE, RAND2 from SU.
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct RandomChallenge {
        pub bytes: [u8; 5],  // 40 bits
    }

    impl RandomChallenge {
        /// Expand from 40 bits to 128 bits by zero-padding LSB side (AM2/AM4 input).
        pub fn expand(&self) -> [u8; 16] {
            let mut out = [0u8; 16];
            out[0..5].copy_from_slice(&self.bytes);
            out
        }
    }

    /// Authentication Response -- 32-bit truncated AES output.
    /// RES1 from SU, RES2 from FNE.
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct AuthResponse {
        pub value: u32,  // 32 bits = MSB 32 of AES-128-ECB output
    }
}
```

### 12.3 TSBK Authentication Parser

```rust
impl AuthMessage {
    /// Parse an authentication message from a TSBK payload.
    /// `data` is the 10-byte payload (octets 0-9, CRC already verified/stripped).
    /// `direction` indicates OSP (outbound, FNE->SU) or ISP (inbound, SU->FNE).
    pub fn from_tsbk(data: &[u8; 10], direction: Direction) -> Option<Self> {
        let opcode = data[0] & 0x3F;
        let mfid = data[1];

        match (direction, opcode) {
            (Direction::Osp, 0x31) => {
                // AUTH_DMD -- abbreviated TSBK (no RS)
                let standalone = (data[2] & 0x01) != 0;
                let rand1_u32 = u32::from_be_bytes([data[3], data[4], data[5], data[6]]);
                let target = u16::from_be_bytes([data[8], data[9]]);
                // Abbreviated form carries 32-bit RAND1 (truncated from 40-bit)
                let mut rand1_bytes = [0u8; 5];
                rand1_bytes[0..4].copy_from_slice(&data[3..7]);
                Some(AuthMessage::AuthDemand {
                    mfid,
                    standalone,
                    rs: None,  // RS not present in abbreviated TSBK
                    rand1: auth::RandomChallenge { bytes: rand1_bytes },
                    target: SuAddress::Abbreviated { unit_id: target },
                })
            }

            (Direction::Osp, 0x32) => {
                // AUTH_FNE_RESP
                let res2 = u32::from_be_bytes([data[3], data[4], data[5], data[6]]);
                let target = u16::from_be_bytes([data[8], data[9]]);
                Some(AuthMessage::AuthFneResponse {
                    mfid,
                    res2: auth::AuthResponse { value: res2 },
                    target: SuAddress::Abbreviated { unit_id: target },
                })
            }

            (Direction::Isp, 0x38) => {
                // AUTH_RESP
                let standalone = (data[2] & 0x01) != 0;
                let res1 = u32::from_be_bytes([data[3], data[4], data[5], data[6]]);
                let source = u16::from_be_bytes([data[8], data[9]]);
                Some(AuthMessage::AuthResponse {
                    mfid,
                    standalone,
                    res1: auth::AuthResponse { value: res1 },
                    source: SuAddress::Abbreviated { unit_id: source },
                })
            }

            (Direction::Isp, 0x3A) => {
                // AUTH_FNE_RST
                let fne_passed = (data[2] & 0x80) != 0;
                let standalone = (data[2] & 0x01) != 0;
                let source = u16::from_be_bytes([data[8], data[9]]);
                Some(AuthMessage::AuthFneResult {
                    mfid,
                    fne_passed,
                    standalone,
                    source: SuAddress::Abbreviated { unit_id: source },
                })
            }

            (Direction::Isp, 0x3B) => {
                // AUTH_SU_DMD
                let source = u16::from_be_bytes([data[8], data[9]]);
                Some(AuthMessage::AuthSuDemand {
                    mfid,
                    source: SuAddress::Abbreviated { unit_id: source },
                })
            }

            _ => None,  // 0x39 (AUTH_RESP_M) is MBT-only, handled separately
        }
    }
}
```

### 12.4 Authentication Mechanism Implementations

```rust
use aes::Aes128;
use aes::cipher::{BlockEncrypt, KeyInit, generic_array::GenericArray};

/// AM1: Derive session authentication key KS from master key K and random seed RS.
/// KS = AES-128-ECB(key=K, plaintext=expand(RS))
pub fn am1(k: &auth::AuthKey, rs: &auth::RandomSeed) -> auth::SessionKey {
    let cipher = Aes128::new(GenericArray::from_slice(&k.bytes));
    let mut block = GenericArray::clone_from_slice(&rs.expand());
    cipher.encrypt_block(&mut block);
    auth::SessionKey { bytes: block.into() }
}

/// AM2: Compute authentication response RES1 from session key KS and challenge RAND1.
/// RES1 = MSB32(AES-128-ECB(key=KS, plaintext=expand(RAND1)))
pub fn am2(ks: &auth::SessionKey, rand1: &auth::RandomChallenge) -> auth::AuthResponse {
    let cipher = Aes128::new(GenericArray::from_slice(&ks.bytes));
    let mut block = GenericArray::clone_from_slice(&rand1.expand());
    cipher.encrypt_block(&mut block);
    let value = u32::from_be_bytes([block[0], block[1], block[2], block[3]]);
    auth::AuthResponse { value }
}

/// AM3: Derive mutual session key KS' from master key K and random seed RS.
/// KS' = AES-128-ECB(key=K, plaintext=complement(expand(RS)))
pub fn am3(k: &auth::AuthKey, rs: &auth::RandomSeed) -> auth::MutualSessionKey {
    let cipher = Aes128::new(GenericArray::from_slice(&k.bytes));
    let mut block = GenericArray::clone_from_slice(&rs.expand_complement());
    cipher.encrypt_block(&mut block);
    auth::MutualSessionKey { bytes: block.into() }
}

/// AM4: Compute mutual authentication response RES2 from mutual session key KS' and RAND2.
/// RES2 = MSB32(AES-128-ECB(key=KS', plaintext=expand(RAND2)))
pub fn am4(ks_prime: &auth::MutualSessionKey, rand2: &auth::RandomChallenge) -> auth::AuthResponse {
    let cipher = Aes128::new(GenericArray::from_slice(&ks_prime.bytes));
    let mut block = GenericArray::clone_from_slice(&rand2.expand());
    cipher.encrypt_block(&mut block);
    let value = u32::from_be_bytes([block[0], block[1], block[2], block[3]]);
    auth::AuthResponse { value }
}
```

### 12.5 Authentication Event Tracking (Decoder State)

```rust
use std::collections::HashMap;
use std::time::Instant;

/// Tracks authentication transactions observed by a passive decoder.
#[derive(Debug)]
pub struct AuthTransaction {
    pub target_id: u16,
    pub rs: Option<[u8; 10]>,    // 80-bit Random Seed (if MBT form seen)
    pub rand1: Option<[u8; 5]>,  // 40-bit RAND1
    pub res1: Option<u32>,       // 32-bit response from SU
    pub is_mutual: bool,
    pub rand2: Option<[u8; 5]>,  // 40-bit RAND2 (mutual only)
    pub res2: Option<u32>,       // 32-bit response from FNE (mutual only)
    pub fne_passed: Option<bool>, // R2 result (mutual only)
    pub standalone: bool,
    pub started_at: Instant,
    pub outcome: AuthOutcome,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum AuthOutcome {
    Pending,
    SuPassed,
    SuFailed,
    FnePassed,
    FneFailed,
    Timeout,
}

pub struct AuthTracker {
    /// Active transactions keyed by target unit ID
    pending: HashMap<u16, AuthTransaction>,
    /// Completed transactions (ring buffer or vec)
    completed: Vec<AuthTransaction>,
    /// Authentication timer duration
    timeout_secs: f64,
}

impl AuthTracker {
    pub fn new() -> Self {
        Self {
            pending: HashMap::new(),
            completed: Vec::new(),
            timeout_secs: 30.0,
        }
    }

    pub fn on_message(&mut self, msg: &AuthMessage, timestamp: Instant) {
        match msg {
            AuthMessage::AuthDemand { target, standalone, rs, rand1, .. } => {
                let target_id = match target {
                    SuAddress::Abbreviated { unit_id } => *unit_id,
                    SuAddress::Full { unit_id, .. } => *unit_id as u16,
                };
                self.pending.insert(target_id, AuthTransaction {
                    target_id,
                    rs: rs.map(|r| r.bytes),
                    rand1: Some(rand1.bytes),
                    res1: None,
                    is_mutual: false,
                    rand2: None,
                    res2: None,
                    fne_passed: None,
                    standalone: *standalone,
                    started_at: timestamp,
                    outcome: AuthOutcome::Pending,
                });
            }
            AuthMessage::AuthResponse { source, res1, .. } => {
                let id = extract_unit_id(source);
                if let Some(tx) = self.pending.get_mut(&id) {
                    tx.res1 = Some(res1.value);
                }
            }
            AuthMessage::AuthResponseMutual { source, res1, rand2, .. } => {
                let id = extract_unit_id(source);
                if let Some(tx) = self.pending.get_mut(&id) {
                    tx.res1 = Some(res1.value);
                    tx.rand2 = Some(rand2.bytes);
                    tx.is_mutual = true;
                }
            }
            AuthMessage::AuthFneResponse { target, res2, .. } => {
                let id = extract_unit_id(target);
                if let Some(tx) = self.pending.get_mut(&id) {
                    tx.res2 = Some(res2.value);
                }
            }
            AuthMessage::AuthFneResult { source, fne_passed, .. } => {
                let id = extract_unit_id(source);
                if let Some(mut tx) = self.pending.remove(&id) {
                    tx.fne_passed = Some(*fne_passed);
                    tx.outcome = if *fne_passed {
                        AuthOutcome::FnePassed
                    } else {
                        AuthOutcome::FneFailed
                    };
                    self.completed.push(tx);
                }
            }
            AuthMessage::AuthSuDemand { .. } => {
                // SU demand is informational -- the AUTH_DMD that follows
                // will create the actual transaction entry
            }
        }
    }

    /// Call periodically to expire transactions older than 30 seconds.
    pub fn expire_stale(&mut self, now: Instant) {
        let timeout = std::time::Duration::from_secs_f64(self.timeout_secs);
        let expired: Vec<u16> = self.pending.iter()
            .filter(|(_, tx)| now.duration_since(tx.started_at) > timeout)
            .map(|(id, _)| *id)
            .collect();
        for id in expired {
            if let Some(mut tx) = self.pending.remove(&id) {
                tx.outcome = AuthOutcome::Timeout;
                self.completed.push(tx);
            }
        }
    }
}

fn extract_unit_id(addr: &SuAddress) -> u16 {
    match addr {
        SuAddress::Abbreviated { unit_id } => *unit_id,
        SuAddress::Full { unit_id, .. } => *unit_id as u16,
    }
}
```

## 13. Cross-Reference Summary

### 13.1 TIA-102.AABC-E (Trunking Control Channel Messages)

Defines the wire-level TSBK and MBT formats for all six authentication messages.
Key reference points:
- Section 6.1.19-6.1.24: ISP authentication message formats (AUTH_FNE_RST, AUTH_RESP, AUTH_RESP_M, AUTH_SU_DMD)
- Section 6.2.25-6.2.26: OSP authentication message formats (AUTH_DMD, AUTH_FNE_RESP)
- Annex B: Deny reason codes 0x22 (SU failed auth), 0x23 (FNE failed auth), 0x24 (SU could not be authenticated)
- Opcode table: ISP 0x38-0x3B, OSP 0x31-0x32

### 13.2 TIA-102.AABF-D (Link Control Word)

LCW opcode 0x12 (Unit Authentication Command) is marked OBSOLETE/RESERVED. Authentication
does not use LCW on traffic channels in the current standard. All authentication occurs
on the control channel.

### 13.3 TIA-102.BBAD-A (TDMA MAC Messages)

TDMA systems map authentication messages into MAC PDU partitions 0b01 (abbreviated)
and 0b11 (extended). The MCO values match the FDMA opcodes exactly. Authentication
for TDMA systems still occurs on the FDMA control channel; the TDMA MAC mappings
exist for systems that might carry signaling on TDMA channels.

### 13.4 TIA-102.AAAD-B (Block Encryption Protocol)

Authentication (AACE) and encryption (AAAD) are independent services with separate
key hierarchies. Authentication protects network access (who can use the system);
encryption protects content (what is said). Both may be enabled simultaneously.
ALGID and KID fields in encrypted traffic are unrelated to authentication K, KS, KS'.

### 13.5 SDRTrunk / OP25 Auth Handling

Open-source P25 decoders handle authentication messages as follows:

**SDRTrunk:**
- Parses AUTH_DMD, AUTH_RESP, AUTH_RESP_M, AUTH_FNE_RESP, AUTH_FNE_RST, AUTH_SU_DMD
  from TSBK streams
- Displays authentication events in the events table (unit ID, pass/fail)
- Does not perform actual authentication computation (no K/KS held)
- Look for `Authentication` classes in the TSBK message decoder package

**OP25:**
- Decodes authentication opcodes in `tsbk.py` / `p25_tsbk.cc`
- Logs authentication events (RAND1, RES1, target ID)
- Does not hold authentication keys or verify responses

Both decoders treat authentication messages as informational metadata -- they extract
and display the observable fields (RAND1, RES1, RES2, R2, addresses) without
performing cryptographic verification.
