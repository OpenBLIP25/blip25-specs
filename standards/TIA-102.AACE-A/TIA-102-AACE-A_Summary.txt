TIA-102.AACE-A: Project 25 Digital Land Mobile Radio — Link Layer Authentication
Summary
Published: April 2011 | Classification: ALGORITHM + PROTOCOL | ~1,000 words

WHAT THIS DOCUMENT SPECIFIES

This document defines the Challenge and Response Authentication service for Project 25
(P25) trunked radio systems. It specifies how a Subscriber Unit (SU) proves its identity
to the Fixed Network Equipment (FNE), and optionally how the FNE proves its identity
back to the SU (mutual authentication). The authentication operates exclusively on the
FDMA trunking control channel and applies to both FDMA and TDMA trunking systems
that use an FDMA control channel. If a system implements either unit authentication or
mutual authentication, it must do so exactly as specified in this document.

WHY THIS DOCUMENT EXISTS

Radio systems used by public safety agencies are vulnerable to rogue devices that
impersonate legitimate subscribers. Without authentication, an adversary SU can register
on the system, consume resources, and potentially intercept traffic. Without mutual
authentication, a rogue base station (adversary FNE) can trap SUs into associating with
it. This document establishes a cryptographically sound challenge-response protocol that
prevents both threats while remaining implementable within the tight bandwidth and
message-format constraints of a P25 trunking control channel.

WHAT IT DEFINES

The document defines authentication at several levels:

1. KEY HIERARCHY AND ROLES
   - Authentication Key K (128 bits): the long-term secret, unique per SUID, stored
     in non-volatile memory in both the SU and the Authentication Facility (AF).
     K never transits the air interface.
   - Random Seed RS (80 bits): generated fresh by the AF; used to derive session keys
     so K is never exposed to the RFSS.
   - Session Authentication Key KS (128 bits): derived by AM1; given to the RFSS to
     enable it to authenticate SUs without possessing K.
   - Mutual Session Authentication Key KS' (128 bits): derived by AM3; given to the
     RFSS to enable SUs to authenticate the RFSS.
   - RAND1 / RAND2 (40 bits each): per-transaction random challenges generated by the
     RFSS (RAND1) and the SU (RAND2).
   - RES1 / XRES1 / RES2 / XRES2 (32 bits each): computed responses and expected
     responses, compared to determine authentication pass or fail.

2. FOUR AUTHENTICATION MECHANISMS (AES-128 ECB)
   All four mechanisms are instances of AES-128 in Electronic Codebook (ECB) mode:

   AM1(K, RS) -> KS
     RS is zero-padded from 80 bits to 128 bits (append 48 zero bits to the LSB side),
     then encrypted with K as the AES key. Output is KS.

   AM2(KS, RAND1) -> RES1
     RAND1 is zero-padded from 40 bits to 128 bits (append 88 zero bits), then
     encrypted with KS as the AES key. The 128-bit ciphertext is truncated to its
     most significant 32 bits to yield RES1.

   AM3(K, RS) -> KS'
     Same as AM1, but the 128-bit expanded RS is bitwise-complemented before
     encryption. This ensures KS != KS' even with the same RS and K.

   AM4(KS', RAND2) -> RES2
     Same as AM2 but uses KS' as the AES key and RAND2 as the plaintext.
     Output is RES2 (32 bits, MSB of 128-bit ciphertext).

3. STATE MACHINES
   Two state diagrams are provided: RFSS focus (for network-side implementation)
   and SU focus (for subscriber-side implementation). Key RFSS states are Idle,
   Want to Authenticate, Authentication Procedure in Progress, and Authenticated SU.
   Key SU states are Idle, Change Site, Authentication Procedure in Progress, and
   Authenticated FNE. The document specifies the guard conditions and transitions
   for adversary resilience: a failing adversary SU must not block service to a valid
   SU; a failing or unresponsive FNE triggers a site-change rather than indefinite
   retry.

4. AUTHENTICATION RULES (11 rules for unit auth, 7 for mutual auth)
   Notable rules include: a 30-second default Authentication Timer that both sides
   start simultaneously; a SU must not submit a separate service request until it
   has sent its authentication response; if RS/KS is unavailable, the RFSS may
   permit or deny service per local security policy; the FNE must be capable of
   mutual authentication; an SU configured for mutual authentication must invoke it.

5. MESSAGE SEQUENCE CHARTS
   Ten MSCs are provided covering both standalone (outside registration) and
   registration-embedded authentication flows, for both pass and fail outcomes in
   unit-only and mutual modes. A "SU Demand" MSC covers the case where the SU
   demands authentication after the RFSS initially declines.

6. CONTROL CHANNEL MESSAGES
   Six messages are required, defined by reference to TIA/EIA-102.AABC:
   Authentication Demand OSP, Authentication Response ISP,
   Authentication Response Mutual ISP, Authentication FNE Response OSP,
   Authentication FNE Result ISP, Authentication SU Demand ISP.

7. KEY MANAGEMENT AND PROVISIONING
   K is provisioned once per SUID via Key Fill Device (per TIA-102.AACD) or OTAR
   (per TIA-102.AACA). The AF derives KS and KS' using RS and distributes them to
   the RFSS, so K never reaches the RFSS. RS rotation (and thus KS/KS' rotation)
   provides crypto-period management without re-keying K. Systems without OTAR must
   not transmit K over the air.

KEY PARAMETERS AND REQUIREMENTS

  Parameter  | Size   | Notes
  -----------+--------+----------------------------------------
  K          | 128 b  | Long-term secret, per-SUID, non-volatile
  KS, KS'    | 128 b  | Session keys, derived by AF, given to RFSS
  RS         | 80 b   | Random seed, AF-generated, per session
  RAND1/2    | 40 b   | Per-transaction random challenges
  RES1/2     | 32 b   | Truncated AES output on air interface
  XRES1/2    | 32 b   | Expected response, computed by RFSS/SU
  Auth Timer | 30 s   | Default; same value used by both SU and RFSS
  AES mode   | ECB    | AES-128, FIPS 197 / NIST SP 800-38A

layer that complements traffic encryption: encryption protects content; this
document protects network access.

LIMITATIONS AND NOTES

- ECB mode is used intentionally: each authentication event produces an independent
  ciphertext block; there is no IV and no chaining. The 30-second timer and
  per-transaction RAND values prevent replay.
- The 32-bit truncation of RES provides only modest response space (2^32). The
  design relies on the secrecy of KS (which requires protecting RS) rather than
  response-space size.
- Reference [4] is blank in the published document — likely a placeholder never
  filled.
- The document was revised eight times between 2003 and 2010 before publication,
  with corrections to AM2/AM4 sample data in revision 8 (March 2005). Implementors
  should verify test vectors against this -A revision.
- TDMA systems are in scope but use the FDMA control channel for authentication;
  no TDMA-specific modifications to the algorithm or protocol are defined.
- The document is published under TIA-102.AACE-A (the -A suffix indicating this
  first revision supersedes the original TIA 102.AACE from September 2005).
