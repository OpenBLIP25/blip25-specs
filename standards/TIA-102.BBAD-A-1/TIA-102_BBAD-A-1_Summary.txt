SUMMARY: TIA-102.BBAD-A-1 — Project 25 Two-Slot TDMA MAC Layer Messages, Addendum 1: Location on PTT and User Alias

DOCUMENT OVERVIEW

TIA-102.BBAD-A-1 is a March 2025 addendum to TIA-102.BBAD-A, the base standard that defines MAC (Medium Access Control) layer messages for the P25 Phase II two-slot TDMA air interface. Published by the Telecommunications Industry Association under the cognizance of the TR-8.10 Air Interface Subcommittee, this addendum introduces three new MAC-layer messages that enable two features not present in the base standard: Location on PTT (transmitting a subscriber unit's GPS coordinates when the push-to-talk button is pressed) and User Alias (transmitting a human-readable name for the talking user). The addendum was approved for publication in January 2025.

WHY IT EXISTS

P25 Phase II TDMA systems already supported voice and data communication, trunking control, and encryption signaling over their two-slot channel structure. However, the base BBAD-A standard did not define a mechanism for subscriber units to embed GPS location data directly into the MAC signaling during a voice call, nor did it provide a way to broadcast an alphanumeric user alias alongside the numeric subscriber ID. Both capabilities are increasingly demanded by public safety agencies for situational awareness: dispatchers and fellow officers need to see where a transmitting unit is located the moment they key up, and they benefit from seeing a plain-text name or callsign rather than just a numeric radio ID. This addendum fills those gaps by standardizing the message formats so that equipment from different manufacturers can interoperate when delivering these features over the TDMA air interface.

WHAT IT DEFINES

The addendum adds three new entries to the VCH (Voice Channel) MAC message tables and the MAC message length tables in the base standard. All three messages are 17 octets long and use manufacturer-specific (MFID) opcodes within the MAC-specific message category. The three messages are:

1. MAC_MFIDA4_UA (Section 3.99 — User Alias): An outbound-only message sent by the system on the SACCH (Slow Associated Control Channel) during an active TDMA voice call. It carries a 14-character ASCII string representing the talking user's alias. The manufacturer ID is $A4, the MCO (MAC opcode) is %101000, and B1/B2 are set to %10. The message occupies octets 4 through 17 with one ASCII character per octet, for a maximum alias length of 14 characters. This message is transmitted by the infrastructure (outbound) so that receiving subscriber units can display the alias.

2. MAC_MFIDA4_LOP (Section 3.100 — MFIDA4 Location on PTT): An inbound-only message sent by the subscriber unit to the system during a PTT event. It uses the same manufacturer ID $A4 but MCO %101010. The 17-octet payload encodes latitude (4 octets), longitude (4 octets), time (2 octets), course and speed (3 octets), and a status byte. Latitude and longitude are encoded in a degrees-minutes format: integer minutes as a 7-bit unsigned integer, decimal minutes as a 16-bit unsigned integer representing ten-thousandths of a minute, and a hemisphere bit (MSB of octet 6 for lat, MSB of octet 10 for lon). Time is represented as a 17-bit count of seconds since midnight, split across the 2-octet Time field (lower 16 bits) and the T bit in octet 17 (17th bit). Course is a 12-bit value in tenths of degrees, and speed is a 12-bit value in tenths of knots, packed across three octets. The status byte encodes emergency state (1 bit), GPS fix quality (2 bits with values for No Fix, GPS Fix, Differential GPS Fix, and No GPS Hardware), and satellite count (4 bits).

3. MAC_MOT_PTT_LOC (Section 3.101 — MFID90 MAC PTT Location): A second inbound Location-on-PTT message, this one using Motorola's manufacturer ID $90 and MCO %100001. It is used during group calls or MFID90 supergroup calls. Its 17-octet structure includes a type field (currently only Point-2D is defined), a direction-valid bit, emergency and encryption bits, a 3-octet group/supergroup address (WGID), a 3-octet source address (WUID), 3-octet latitude, 3-octet longitude, and packed minute/second/direction/speed fields. The latitude and longitude encoding here differs from the MFIDA4 variant: it uses the LRRP (Location Request Response Protocol) encoding where latitude is computed as (L * 2^31) / 90 in a sign-plus-31-bit format, and longitude is computed as (L * 2^32) / 360 in 32-bit two's complement, then truncated to three bytes with rounding. Direction is quantized to 8 compass points (45-degree resolution). Speed is quantized into 2 m/s bins with 5 bits, capping at 60 m/s (approximately 134 mph). The time field provides only minute (6 bits) and a coarse second value (2 bits encoding 0/15/30/45 seconds).

KEY TECHNICAL PARAMETERS

All three messages are fixed at 17 octets in length. They ride on the SACCH within the TDMA voice channel structure, meaning they are embedded in the signaling that accompanies active voice traffic rather than requiring a separate data channel. The two Location-on-PTT messages represent two different manufacturer encodings for essentially the same feature: MFIDA4 ($A4) uses a degrees-minutes coordinate format with high-resolution time, course, and speed, while MFID90 ($90) uses the LRRP-derived binary encoding with coarser time and speed resolution but adds explicit group address and source address fields. Both are inbound (subscriber-to-infrastructure), while the User Alias message is outbound (infrastructure-to-subscriber).

The MFIDA4 location message provides finer granularity for time (1-second resolution via the full 17-bit seconds-since-midnight field), speed (tenths of knots), and course (tenths of degrees). The MFID90 variant trades precision for compactness in certain fields: time has only 15-second resolution, direction has 45-degree resolution, and speed has 2 m/s bins. However, the MFID90 message carries addressing information (group and source) directly in the message body, which the MFIDA4 variant does not.

LIMITATIONS

The document explicitly acknowledges that some aspects of its specifications may not have been fully operationally tested. The User Alias is limited to 14 ASCII characters — sufficient for short callsigns or badge numbers but potentially limiting for full names. The MFID90 location encoding currently defines only Point-2D (lat/lon without altitude); other types are reserved for future use. The addendum does not define any new control-channel messages, acknowledgment mechanisms, or error-handling procedures for these features — it covers only the MAC-layer message formats themselves. Implementers must refer to the base TIA-102.BBAD-A standard for all surrounding protocol machinery, and to TIA-102.BAJC for LRRP encoding details.

The addendum's section numbers (3.99, 3.100, 3.101) are designed to slot directly into the base standard's numbering, and the next full revision of TIA-102.BBAD-A is expected to incorporate these additions.
