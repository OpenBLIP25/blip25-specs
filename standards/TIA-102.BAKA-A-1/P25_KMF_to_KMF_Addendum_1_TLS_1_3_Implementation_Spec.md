## 1. Why This Addendum Exists

TIA-102.BAKA-A (parent) originally specified **only TLS 1.2** for
optional transport protection of the KMF real-time transport mode.
TLS 1.2 has known weaknesses vs TLS 1.3 — notably:

- **Forward secrecy** — TLS 1.3 makes ephemeral key exchange
  mandatory; TLS 1.2 allowed non-FS modes.
- **Cipher agility** — TLS 1.3 dropped many legacy ciphers.
- **Handshake complexity** — TLS 1.3 simplified / modernized.

This addendum adds TLS 1.3 support without forcing new deployments
to abandon TLS 1.2 backward compatibility with older KMFs.

## 2. Two KMF-to-KMF Transport Methods

Per BAKA-A §2. **Both are required** for conforming KMFs:

| Method | Transport | Security |
|--------|-----------|----------|
| **Non-real-time** | S/MIME files via email / FTP / USB | S/MIME only (no TLS) |
| **Real-time** | S/MIME messages over TCP | S/MIME + **optional** TLS |

**Return receipts** are generated by the destination KMF after
successful processing. If receipts cannot be delivered, the
originating KMF operator may manually override pass / fail status.

**TLS provides** (per BAKA-A §2.4.3 as clarified by this addendum):
- Replay protection (distinct from S/MIME's content protection).
- A second security layer on top of the S/MIME encryption.
- Mutual authentication (required when TLS is used).
- RSA key exchange (required when TLS is used).

## 3. Cipher Suite Profiles

Per BAKA-A-1 §A.4 (modifies parent §3.2):

### 3.1 TLS 1.2 (mandatory if TLS supported)

```
Cipher suite: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 [RFC5289]
RSA key size: 3072 bits (X.509 certificate)
Symmetric cipher: AES-256-GCM
MAC: HMAC-SHA-384
Key exchange: ECDHE with RSA auth
```

### 3.2 TLS 1.3 (optional; added by this addendum)

```
Cipher suite: TLS_AES_128_GCM_SHA256 [RFC8446]
RSA key size: 4096 bits (X.509 certificate)
Symmetric cipher: AES-128-GCM
MAC: HMAC-SHA-256 (both KDF and handshake MAC)
Key exchange: TLS 1.3 native ephemeral (RSA used for certificate auth)
```

### 3.3 Other cipher suites

Permitted **by administrative agreement** between KMF operators.
The above two are the standardized defaults; a bilaterally-configured
deployment may use stronger (or differently-tuned) suites if both
operators agree.

## 4. X.509 Certificate Requirements

Per BAKA-A-1 §A.5 (modifies parent §3.3). Each KMF needs **up to three
distinct certificates**:

| Purpose | Required | Key size | Notes |
|---------|----------|----------|-------|
| **S/MIME RSA encryption** | Always | Per S/MIME profile | Outer / inner layer key material wrapping |
| **Digital signatures only** | Always | Per S/MIME profile | For message signatures |
| **TLS layer authentication** | Only if TLS deployed | **3072 (TLS 1.2) or 4096 (TLS 1.3)** | **Must be distinct from the S/MIME certificates** |

**Dual-purpose certificate** — a single cert may handle both signing
and encryption for **all S/MIME operations** as an optional
simplification. The TLS cert remains separate.

**Certificate framework:** all certificates conform to **RFC 5280**
(Internet X.509 PKI Certificate and CRL Profile). Cross-CA and
subordinate-CA structures are supported.

**KMF identification:** no specific KMF-ID attribute is required in
end-entity certificates. KMF identification may use the Common Name
(CN) or other Subject field as needed by the operator.

## 5. Connection Establishment

Per BAKA-A-1 §A.6 (modifies parent §4.2.3.1):

- **KMF as TCP server:** listens on **TCP port 49199** (per
  TIA-102.BAJD / BAJD-A) or an administratively-agreed alternative.
- **Connection procedure:** standard TCP per RFC 793.
- **TLS handshake:** if TLS is deployed (by administrative agreement),
  follows immediately after TCP establishment.
  - The **TCP connection initiator becomes the TLS client**.
  - The **TCP acceptor becomes the TLS server**.
- **Interop fallback:** KMFs **MUST be able to operate without TLS**
  to ensure base interoperability.

Read BAKA-A for these (unchanged):

- **S/MIME message formats** — parent §2, §3.
- **OTAR command encoding** — parent references AACA series.

- **Conformance test cases** — parent conformance chapter.

**Also NOT covered by this addendum:**
- Conformance test cases **for TLS 1.3** — out of scope; would need
  separate work.
- Cipher suite enumeration beyond the two standardized defaults.
- Key-management policy between KMF operators (agency-specific).
