## §8.2.1 Codeword layout

`ENCODING_TABLE_24128[data]` returns a 24-bit codeword in this layout:

```
Bit:  23 ............ 12 | 11 ........... 0
      [ 12-bit info     ] | [ 12-bit parity ]
```

- High 12 bits (positions 12–23) = systematic info word (= `data`).
- Low 12 bits (positions 0–11) = 11 bits of (23,12,7) BCH parity at
  positions 1–11, plus the extended overall-parity bit at position 0.

This matches the impl spec §8.2.1 layout: each generator-row entry
is `[12-bit identity | 12-bit parity]` packed into the low 24 bits of
a `uint32_t`.

For data = 2^k (single info bit at position k), the codeword is
exactly the k-th generator-matrix row:

```
codeword = (1 << (12 + k))  |  parity_for_info_bit_k
```

The 12 entries in the verification table above are precisely
`ENCODING_TABLE_24128[2^k]` for k = 0…11.

## Verification methodology

```python
import re

# Extract the 4096 entries from MMDVMHost ENCODING_TABLE_24128.
in_table = False
entries = []
with open('Golay24128.cpp') as f:
    for line in f:
        if 'ENCODING_TABLE_24128' in line and '=' in line:
            in_table = True
            continue
        if in_table:
            if '};' in line:
                break
            for m in re.findall(r'0x([0-9A-Fa-f]+)U?', line):
                entries.append(int(m, 16))

# Compare against impl spec §8.2.1 GOLAY_24_12_GENERATOR rows.
spec = {
    1:    0x0018EB,  # row 12 (info bit 0)
    2:    0x00293E,  # row 11
    4:    0x004A97,  # row 10
    8:    0x008DC6,  # row 9
    16:   0x010367,  # row 8
    32:   0x0206CD,  # row 7
    64:   0x040D99,  # row 6
    128:  0x0803DA,  # row 5
    256:  0x1007B4,  # row 4
    512:  0x200F68,  # row 3 ← previously "verify"
    1024: 0x40063B,  # row 2
    2048: 0x800C75,  # row 1 (info bit 11)
}
for k, v in spec.items():
    assert entries[k] == v, f"mismatch at data={k}"
print("all 12 generator rows match MMDVMHost ENCODING_TABLE_24128")
```

This confirms the impl spec's full generator matrix at the codeword
level. Because the (24,12,8) code is linear, every other entry in the
4096-entry table is determined by XOR'ing the appropriate subset of
these 12 rows, so verifying the rows verifies the table.

## Validity check on row 3 (the "verify" entry)

Row 3's codeword `0x200F68` for data = 512 (= info bit 9 set, all
others zero):

- Info portion (bits 23–12): `0x200` = `0010 0000 0000` → bit 9 set
  → matches data = 512. ✓
- Parity portion (bits 11–0): `0xF68` = `1111 0110 1000`.
- Total bit weight: popcount(`0x200F68`) = 1 + 7 = 8. The minimum
  Hamming weight of any non-zero (24,12,8) codeword is 8 (= d_min for
  the extended code), so this codeword sits at the minimum-weight
  boundary and is consistent with a generator-row entry.
- The (23,12,7) form of the same codeword (drop bit 0): `0x200F68 ^
  bit_0 = 0x200F68` itself if bit 0 is 0 (it is — last hex digit `8`
  has bit 0 clear). So `table_23127[512] = 0x200F68 >> 1 …` is not the
  correct relationship; rather, MMDVMHost stores both forms with info
  in the same high 12 bits and (23,12)-parity at bits 1–11, with the
  extended bit at position 0. For row 3 the extended bit is 0
  (already even parity in the (23,12) part), so `table_24128[512] ==
  table_23127[512]`. Both tables agree on this row.

The "verify" caveat (and the broader "GOLAY_24_12_GENERATOR -- verify
against reference implementation" comment in §8.2.1) can be lifted.

## Encoder / decoder algorithms

Both functions are precomputed-table lookups at the API surface:

- `encode24128(data)` → `ENCODING_TABLE_24128[data]` (line 1080).
- `decode24128(in, &out)` (line 1093):
  1. Drop the extended parity bit: `c23 = in >> 1`.
  2. Compute the (23,12,7) syndrome via `get_syndrome_23127(c23)` —
     polynomial division by `GENPOL = 0xC75`.
  3. Look up the error pattern: `e = DECODING_TABLE_23127[syndrome] << 1`.
  4. Correct: `corrected = in ^ e`.
  5. Validity check: `valid = (popcount(syndrome) < 3) || (popcount(corrected) is even)`.
  6. Extract info: `out = corrected >> 12`.
  7. Return `valid`.

The validity test (step 5) is the standard extended-Golay
disambiguation: a (23,12,7) syndrome with weight ≥ 3 typically
indicates 4 errors that the (23,12,7) decoder will silently
mis-correct; in the (24,12,8) form, the extended parity bit catches
these by forcing odd total weight. The ` || (popcount(corrected) is
even)` condition accepts the correction when even parity is preserved
(consistent with ≤ 3 errors corrected).

## Used by P25Data.cpp via two interfaces

`P25Data.cpp` calls into Golay24128 along two paths:

| Use site | P25 frame | Code form | MMDVMHost path |
|---|---|---|---|
| `encodeHeaderGolay` / `decodeHeaderGolay` (lines 631–688) | HDU | Golay(18,6,8) shortened | zero-pad 6 info bits to 12, call (24,12,8) encode, take low 18 bits; reverse for decode |
| (Not in this file — see P25 TDULC handler) | TDULC | Golay(24,12,8) full | direct (24,12,8) encode/decode on each 12-bit info pair |

The §8.2.2 implementation-shortcut note added in the
`baaa-b-impl-spec-audit-patches` branch documents this directly: a
single (24,12,8) implementation handles both. This audit confirms the
table backing that single implementation matches the spec for all 12
generator rows, so the shortcut is safe to recommend without
qualification.

## What this audit does NOT cover

- **`DECODING_TABLE_23127`** (2048 entries, syndromes → error
  patterns). The decoder's correction floor (3 errors for (23,12,7),
  3 errors plus extended-bit ambiguity-resolution for (24,12,8)) is
  determined by the contents of this table, not by the generator
  matrix. Bit-level audit deferred. The encoder-side validation here
  is sufficient to confirm the spec's generator matrix; the decoder
  is mathematically determined by the same generator and any
  conformant Berlekamp-style decoder will recover the same
  information.
- **`ENCODING_TABLE_23127`** (the (23,12,7) base table, 4096 entries).
  Used internally by `encode23127`; not invoked by P25Data.cpp on
  encode paths (which use `encode24128` directly). Checking it would
  be redundant given the (24,12,8) verification.
- **The shortened (18,6,8) HDU usage** is structurally validated in
  `analysis/p25_hdu_ldu_lsd_implementation_audit.md`. This audit's
  result implies that the HDU shortened encoding, which goes through
  the same `ENCODING_TABLE_24128` with zero-padded inputs, also
  produces correct codewords.
